# Implement English + Finnish Language Support

## Objective

Implement a proper internationalization (i18n) system for the existing Café Berry website.

The website is already built as a single-page static HTML application with custom JavaScript. We are **not** rebuilding or restructuring the site.

The goal is to layer multilingual support on top of the existing architecture while changing as little of the current HTML, CSS and JavaScript as possible.

English remains the default language.

Finnish is the second supported language.

The implementation must be scalable so additional languages can be added later without requiring HTML changes.

---

# Existing Site Architecture

The website already contains:

* Single-page layout
* Sticky navigation
* Desktop navigation
* Mobile overlay navigation
* Hero section
* Gallery
* Dynamic gallery/lightbox
* Quote/interlude section
* Large menu system
* Sticky menu rail
* Visit section
* Newsletter
* Footer
* Existing JavaScript interactions
* Responsive/mobile layouts

The language system must integrate into the existing structure without breaking any of these features.

Do **not** rewrite or duplicate components.

---

# Do NOT Create Multiple HTML Files

Do not create:

```
/en/index.html

/fi/index.html
```

Do not duplicate the website.

Maintain one HTML version only.

Everything should be translated dynamically.

---

# Translation Files

Create a dedicated translations directory.

Example:

```
/lang

    en.json

    fi.json
```

Every user-facing string should exist inside these files.

Nothing user-visible should remain hardcoded unless it is intentionally language-independent (prices, numbers, IDs, image filenames, etc.).

---

# Translation Keys

Use nested keys grouped by section.

Example:

```
nav.*

hero.*

gallery.*

quote.*

menu.*

visit.*

newsletter.*

footer.*

meta.*
```

Avoid one giant flat JSON.

Keep translations organized exactly like the website sections.

---

# HTML

Do not rewrite the HTML structure.

Simply replace visible text with translation attributes.

Example:

Instead of

```html
<h2>Crafted with care</h2>
```

use

```html
<h2
class="section-title"
data-i18n="menu.heading">
</h2>
```

Do this consistently throughout the website.

---

# Translate Attributes

Support translation of attributes.

Examples:

```
alt

title

placeholder

aria-label

aria-description
```

Suggested syntax:

```
data-i18n-alt

data-i18n-title

data-i18n-placeholder

data-i18n-aria-label
```

Examples include:

* logo alt text
* hero image alt
* gallery buttons
* lightbox controls
* hamburger menu
* close button
* newsletter placeholder

Accessibility text should also change language.

---

# Meta Tags

Translate metadata as well.

This includes:

* page title
* meta description
* Open Graph title
* Open Graph description
* Twitter title (if added)
* Twitter description (if added)

When language changes:

Update these dynamically.

Also update

```
<html lang="">
```

between

```
en

fi
```

---

# Structured Data

The Schema.org JSON-LD should also support translation.

Items like:

* description
* cuisine names (if desired)
* business description

should come from the translation files where appropriate.

Business identifiers such as name, address, and opening hours remain unchanged.

---

# Navigation

Both desktop navigation and mobile navigation must always display the same language.

This includes:

Desktop:

```
Gallery

Menu

Visit
```

Mobile overlay:

```
Gallery

Menu

Visit
```

There must never be a situation where one is translated and the other is not.

---

# Hero Section

Translate:

* tagline
* hero title
* emphasis text
* buttons
* scroll indicator

Do not modify animations.

Only replace text.

---

# Gallery

Translate:

* heading
* description
* lightbox buttons
* pagination labels
* accessibility labels
* image alt text where applicable

Do not modify gallery functionality.

---

# Quote Section

Translate:

* quotation
* attribution

Maintain typography exactly as it currently is.

---

# Menu Section

The menu is the largest translation task.

There is already a

```
menu.txt
```

in the project root.

This file already contains Finnish translations for many menu items.

Use this file as the primary translation source.

Do not manually retranslate menu items that already exist.

Everything in the menu should become translation keys.

Translate:

* category titles
* category descriptions
* food names
* food descriptions
* drink names
* drink descriptions
* legends
* labels
* section headings

Do NOT translate:

* prices
* quantities
* measurements
* allergens icons
* IDs

Keep all current HTML layout intact.

---

# Dynamic JavaScript Content

Any strings inserted by JavaScript must also come from translation files.

Examples:

Gallery labels

Lightbox controls

Status messages

Loading text

Error text

Future popups

Never hardcode English inside JavaScript.

---

# Language Switcher

Add a language selector to the desktop navigation.

Recommended:

```
EN

FI
```

Small text button or dropdown.

For mobile:

Include the same language selector inside the mobile overlay menu.

Language switching should work from either location.

---

# Remember User Preference

When the user changes language:

Save

```
localStorage.language
```

On page load:

1. Check localStorage.

If available:

Load that language.

Otherwise:

Use

```
navigator.language
```

If browser starts with

```
fi
```

load Finnish.

Otherwise

load English.

---

# Browser Detection

Supported:

```
fi

fi-FI
```

↓

Finnish

Everything else

↓

English

---

# Loading Behaviour

Only load the selected language JSON.

Do not fetch every language on startup.

Translation files should be cached after first load.

---

# Missing Translation Behaviour

If a key is missing:

Never leave blank text.

Fallback to English.

Also log a console warning identifying the missing key.

---

# Animations

Changing language must NOT:

restart animations

reset gallery

reset lightbox

change scroll position

reload the page

rebuild components

Only text should change.

---

# Performance

Keep translation logic in one file.

Suggested:

```
js/language.js
```

This file should:

* load translations
* replace text
* update attributes
* update metadata
* update HTML lang
* save language
* restore language
* expose a simple `setLanguage(languageCode)` method

Keep all language-related logic isolated from the rest of the project.

---

# Future Languages

Adding another language should only require:

1. Creating a new JSON file.

2. Registering it in the language selector.

No HTML changes should be necessary.

---

# Deliverables

* Translation system integrated into the existing website.
* `js/language.js`.
* `lang/en.json`.
* `lang/fi.json`.
* All visible text converted to translation keys.
* Menu integrated using the existing `menu.txt`.
* Meta tags translated.
* JSON-LD updated where appropriate.
* Desktop and mobile language switchers.
* Browser language detection.
* localStorage persistence.
* English fallback for missing keys.
* No regressions to existing animations, layout, responsiveness, gallery, lightbox, or sticky menu navigation.

The final implementation should feel like multilingual support was built into the site from the beginning, while preserving the current design, performance, and user experience.
